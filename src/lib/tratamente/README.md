# Tratamente — Sursa Canonică

- `stadii-canonic.ts` este sursa unică pentru `StadiuCod`, label-urile RO, categoriile de management și profilurile per `GrupBiologic`.
- Persistența, importul și generatorul folosesc coduri canonice snake_case; UI-ul afișează doar label-urile mapate din acest fișier.
- Compatibilitate: `listStadiiInOrdine()` și `getOrdine()` păstrează profilul implicit Rubus; pentru UX contextual și reguli noi se folosesc `listStadiiPentruGrup(...)`, `getOrdineInGrup(...)` și `getStadiuUrmatorInGrup(...)`.
- `configurare-sezon.ts` adaugă stratul sezonier per parcelă: sistem de conducere pentru Rubus, tip de ciclu pentru solanacee și `getLabelStadiuContextual(...)` pentru label-uri UI dependente de sezon.
- Pentru Rubus, codurile canonice sunt păstrate pentru compatibilitate, iar vocabularul afișat este rafinat prin `getLabelPentruGrup(...)`; în Rubus mixt, cohorta poate contextualiza eticheta fără să schimbe valoarea stocată.
- Fluxul complet al modulului este: stadii canonice → profiluri biologice → categorii de management → cohorte → configurare sezonieră → generator și RPC atomic.
- Faza 4 a adus Rubus mixt cu cohortele `floricane` și `primocane`. Regula de bază rămâne: dacă parcela nu este Rubus mixt, toate câmpurile de cohortă rămân `NULL`; dacă este mixtă, stadiile pot fi înregistrate pe cohortă, liniile de plan pot avea `cohort_trigger`, iar aplicările pot salva `cohort_la_aplicare`.
- Faza 5 a curățat stratul final: anul curent de sezon vine din `getCurrentSezon()`, iar `allowCohortTrigger` se activează doar când `configurareSezon.sistem_conducere === 'mixt_floricane_primocane'` pentru Rubus.
- Schema V2 păstrează `planuri_tratament_linii`, dar linia este semantic o intervenție planificată. Produsele planificate sunt în `planuri_tratament_linie_produse`, iar câmpurile legacy `produs_id` / `produs_nume_manual` / doze de pe linie rămân doar fallback temporar.
- Execuția rămâne în `aplicari_tratament`, cu `sursa = din_plan | manuala`; produsele efectiv folosite sunt în `aplicari_tratament_produse`, cu snapshot de produs/doză pentru istoric și conformitate. Query/UI/import pot face dual-read temporar până la refactorul complet pe V2.
- Data layer-ul citește și scrie V2 în `src/lib/supabase/queries/tratamente.ts`: planurile expun intervenții cu `produse[]`, aplicările expun `produse_aplicare[]`, iar câmpurile legacy sunt hidratate din primul produs pentru compatibilitate temporară cu UI/generator/import V1.
- UI-ul minim V2 pentru editorul de plan tratează linia ca intervenție și editează `produse[]` în wizard și în detaliul planului. Selectarea din bibliotecă sau numele manual sunt suportate per produs; la salvare se păstrează primul produs în câmpurile legacy doar pentru fallback.
- Selectorul comun `ProdusFitosanitarPicker` este piesa reutilizabilă pentru biblioteca tenant-first: apare în editorul de plan, în aplicarea din plan și în intervenția manuală, afișează activele primele, ascunde inactivele implicit și permite quick-create în biblioteca fermei.
- Execuția din plan folosește `MarkAplicataSheet`: pornește din `aplicari_tratament_produse` dacă există, altfel din produsele planificate ale intervenției, permite override minim pe produse și salvează `aplicari_tratament_produse` plus `diferente_fata_de_plan`.
- Relevanța operațională pe fenofază este calculată în `listInterventiiRelevanteParcela(...)` / `listInterventiiRelevanteHub(...)`: doar intervențiile din planul activ, potrivite cu stadiul observat curent, sunt afișate ca relevante. Repetarea MVP suportă `fara_repetare` și `interval`; intervențiile manuale nu acoperă automat o linie din plan, pentru a evita asocieri istorice greșite.
- Excel-ul oficial de import este V2: foile principale sunt `Interventii` și `Produse interventii`, legate prin `interventie_key`. Review-ul importului lucrează cu intervenții și produse copil, iar salvarea trimite `produse[]` către RPC-ul V2. Parserul citește temporar și formatul legacy V1, dar doar ca fallback explicit.
- Regula de implementare pentru stadii noi: adăugăm codul în helper, mapăm label-ul RO în același fișier, extindem profilul biologic relevant și consumăm doar codul canonic în DB/payload/import.
- În UI, stadiile se afișează fie prin `getLabelRo(...)` pentru cazuri strict globale, fie prin `getLabelPentruGrup(...)` / `getLabelStadiuContextual(...)` când există cultură, grup biologic, sezon sau cohortă; nu se compară string literal de stadiu în logică, ci `StadiuCod`.
