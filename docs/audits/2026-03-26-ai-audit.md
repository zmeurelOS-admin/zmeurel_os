# AI Audit — 2026-03-26

Scope: audit only, no runtime code changes.

1. MODELUL REAL DIN RUNTIME
- provider: `@ai-sdk/google`
- model: `AI_GEMINI_MODEL` cu fallback local la `gemini-2.5-flash`; model simplu opțional `AI_GEMINI_SIMPLE_MODEL` pentru cereri scurte
- fișier: `src/app/api/chat/route.ts`
- verdict scurt: adecvat

2. STAREA FRONTEND + CONTRACT
- verdict: problematic
- Backend-ul livrează contract structurat bun: `type: 'form'` și validează `open_form` cu Zod înainte să ajungă în UI.
- `cheltuiala` și `investitie` sunt singurele flow-uri cu handoff complet confirmat până în page client prin `openForm` + query params.
- `recoltare` pierde prefill-ul la navigare: bottom sheet duce userul la `/recoltari`, dar pagina ascultă doar `?add=1`, nu `openForm` și nici câmpurile extrase.
- `activitate` pierde handoff-ul spre dialog, iar confirmarea directă din bottom sheet poate salva fără `parcela`, deși backend-ul o extrage.
- `comanda` și `client` au query params construiți în `AiBottomSheet`, dar page client-urile nu consumă `openForm` și nu deschid formularul precompletat.

3. STAREA BACKEND FLOW + MULTI-TURN
- verdict: parțial
- Continuitatea multi-turn scurtă există real și este confirmată de `npm run test:ai-chat:integration` (6/6 teste verzi în 2026-03-26).
- Sticky flow + merge cu ultimul mesaj + entity locking reduc reseturile pe follow-up-uri ca `ieri`, `20 kg`, `0,5 l`, `Maria`.
- Continuitatea depinde însă de ultimul schimb din același `pathname` și de regex-uri pe ultimul răspuns AI; dacă acel semnal lipsește, follow-up-ul revine la parsare proaspătă.
- Pragul de `required_for_open_form` rămâne deliberat minimalist, deci unele formulare se pot deschide corect tehnic, dar încă prea incomplete semantic pentru fluxul zilnic.
- Editarea deterministă din `route.ts` acoperă acum doar `cheltuiala`, `investitie` și `activitate`; nu acoperă `recoltare`, `comanda`, `client`.

4. STAREA ENTITY RESOLUTION
- verdict: parțial
- Există deja normalizare lowercase + fără diacritice + fără punctuație, prefix matching, typo tolerance mică și clarificare la ambiguitate.
- Matching-ul este legat de date reale ale tenantului, nu de nume hardcodate în prompt.
- Candidatele sunt totuși prea înguste: nu intră explicit `soi`, `cultura`, alias-uri sau display names de parcelă, deși datele există în query-ul de parcele.
- Nu există un contract structurat separat pentru `raw mention`, `canonical match`, `confidence` și `candidate list`; totul rămâne implicit în pipeline.
- Produsele sunt învățate doar din istoric (`comenzi.produs`, `activitati_agricole.produs_utilizat`), nu dintr-un context mai bogat al fermei.

5. STAREA OBSERVAȚIILOR / LEFTOVER TEXT
- verdict: parțial
- Pe financiar, `descriere` nu mai este dump automat de leftover: textul este curățat de sume, date, stop-words și valori deja consumate.
- Nu există încă o regulă globală unică pentru `observatii` pe toate flow-urile; în afara financiarului, curățarea nu este standardizată la același nivel.
- `sanitizePrefill` elimină doar valori goale sau invalide; nu filtrează zgomot gramatical în `observatii`.
- `comanda` mută `sursa` în `observatii` ca text liber (`Sursa: ...`), deci un câmp structural ajunge într-un câmp liber.
- Pe `recoltare` și `activitate`, deterministicul nu aruncă leftover brut în `observatii`, ceea ce e bine; problema rămâne mai ales pe contractul global și pe fallback-ul LLM.

6. RAPORTAREA LA DATELE REALE ALE USERULUI
- verdict: parțial
- Backend-ul folosește deja date reale tenant pentru matching pe parcele, clienți și produse recente.
- LLM-ul nu primește însă o vedere bogată a fermei; contextul către prompt rămâne mai ales la nivel de count-uri și câteva snippet-uri pe keyword.
- Soiurile și culturile reale ale userului nu sunt puse explicit în mecanismul principal de matching, deși există în datele de parcele.
- Nu există ranking contextual după istoric recent relevant al userului pentru a prefera entitățile cele mai probabile.
- Rezultatul actual este: AI-ul lucrează cu date reale, dar prea îngust și prea puțin explicit pentru robustețe zilnică mare.

7. CAUZA PRINCIPALĂ DOMINANTĂ
- combinație cu o cauză dominantă clară
- dominantă clară: mismatch frontend-backend

8. CE NU ESTE PROBLEMA PRINCIPALĂ
- Modelul principal folosit în runtime nu pare blocajul dominant.
- Lipsa autonomiei AI sau a scrierii directe în DB nu este problema reală acum.
- Lipsa de memorie lungă nu este cauza dominantă pentru fluxurile zilnice scurte.
- Lipsa de funcții noi nu este ce ține pe loc valoarea produsului acum.
- Cost optimizations existente nu par cauza principală a ratărilor observate.

9. MERITĂ URCAT MODELUL ACUM?
- nu
- Mai întâi trebuie închis gap-ul dintre extracția corectă din backend și formularul real pe care îl vede userul, apoi merită reevaluat dacă mai rămâne o limitare semantică reală de model.

10. VERDICT DE PRODUS
- valoarea următoare vine mai ales din repararea și rafinarea nucleului existent
- Backend-ul AI este deja mai bun decât experiența finală percepută; câștigul imediat vine din handoff corect, entity resolution mai bogat și igienă mai strictă a payload-ului, nu din funcții noi.

11. URMĂTORII PAȘI CLARI
- 3 pași prioritari pe partea AI
- 1. Extinderea entity resolution pe date reale tenant cu `nume_parcela + soi + cultura + alias/display`, plus clienți și produse recente, cu clarificare bazată pe confidence.
- 2. Introducerea unui strat unic de extraction hygiene: separare explicită între `raw mention`, `canonical value`, `missing required`, `leftover candidate` și `observatii candidate`.
- 3. Extinderea corpusului de regresii pe română reală, fără diacritice, typo mic și ambiguități apropiate pentru cele 3 trasee zilnice critice.
- 3 pași prioritari pe partea manuală / clasică
- 1. Alinierea completă a handoff-ului `open_form` pentru `recoltare`, `activitate`, `comanda`, `client`, astfel încât ce extrage backend-ul să ajungă în dialogul real.
- 2. Legarea valorilor canonice la ID-urile reale cerute de UI acolo unde formularul lucrează cu select-uri (`parcela_id`, `client_id`, apoi ce alte ID-uri sunt obligatorii la salvare).
- 3. Adăugarea unor teste UI/contract pentru traseul `AiBottomSheet -> route page -> dialog prefilled`, pentru că testele actuale se opresc la API.

12. PASUL SINGUR CU IMPACT MAXIM
- primul pas: repararea contractului backend -> UI pentru toate formularele deja suportate
- de ce acesta este primul: backend-ul deja extrage și clarifică mai bine decât reiese din UX, dar în 4 flow-uri critice datele se pierd sau nu se deschid în formularul real; fără acest pas, auditul de acuratețe AI rămâne parțial mascat de un defect de handoff
- ce NU trebuie făcut încă: nu urcăm modelul, nu adăugăm funcții noi, nu refactorizăm mare route-ul AI înainte să existe paritate completă pe handoff
